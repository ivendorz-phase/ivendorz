Design-taste findings

Raised 2026-07-22 against the live landing page. Toggle each finding to compare; severities follow CLAUDE.md §13. Nothing here is applied to production until ruled.

F1MajorEyebrow density
Current
6 uppercase eyebrow labels on 9 sections, all consecutive. The taste rule caps eyebrows at 3 per page, never on adjacent sections.
Proposed
Keep “Start here” and “How it works” only; the other four headlines stand on their own.

Conflicts with the owner reference, which puts an eyebrow on every section head. Reference fidelity is binding, so this one is an owner ruling, not a dev fix.

F2MinorHero trust strip
Current
Three checkmark claims inside the hero. “No fees to post” repeats the panel CTA, “Verified suppliers” repeats the subtext, and “Quotes as scheduled” reads oddly.
Proposed
Remove the strip. The free-to-post and verification claims already live in the FAQ and CTA band.
F3MinorOne label per intent
Current
The same conversion intent carries three labels: “Let’s Start Procurement”, “Post Your RFQ for free”, and “Post an RFQ”. All route to sign-in.
Proposed
“Post an RFQ” everywhere.

“Create RFQ” in the Command Center is a spec-governed label (landing_page_spec §2) and is deliberately NOT changed by this prototype.

F4MinorLogo strip motion
Current
The marquee auto-scrolls six empty placeholder slots: animated emphasis on emptiness.
Proposed
Static row while no permissioned logos exist; the marquee returns with real content.
F5NitStep number badges
Current
Mono “01” to “04” badges beside icons in How-it-works.
Proposed
Icon only; the ordered layout already carries the sequence.
F6ObsHeadline casing
Current
“The Industrial Procurement HUB for Bangladesh” with mid-sentence all-caps.
Proposed
“Hub” in title case.

The corpus tagline reads “Operating System”, a separate wording question for the owner.

F7ObsFreshness claim
Current
“Published-only, always current” on a surface that fetches nothing yet.
Proposed
“Published listings only”.
F8MinorUnverified feature claim
Current
The FAQ promises “recurring RFQs with standing vendor shortlists and scheduled reorders”, features not verified against Doc-4E.
Proposed
“Yes. You can post one-off or repeat requests.” until the feature set is confirmed.
Governance. This prototype coins nothing: copy, seed data, tokens, and layout mirror the live page (with the 2026-07-22 em-dash and honest-claims copy fixes as the Current baseline). F1 and F3 touch reference-fidelity or spec-governed territory and need an owner ruling before implementation. Already applied in the working tree, not shown as findings: em-dash removal across all visible copy, and the FAQ answers that asserted a success-fee pricing model and a quote-timing SLA.
