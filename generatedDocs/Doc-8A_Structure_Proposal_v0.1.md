Two governing rules shape the document:

1. **Realize, never re-decide — and never re-specify.** Doc-2/3/4 fixed *what* the system is (FROZEN); Doc-5/6/7 fixed *how* it is realized (API / DB / FE). Doc-8 realizes *how those are proven correct* — executable suites whose every assertion is an oracle-by-pointer into a frozen declaration. Doc-8 re-decides nothing in Doc-2/3/4 and re-specifies nothing in Doc-5/6/7; a test may never assert behavior the corpus does not declare, nor a stricter/looser rule than the owning contract.
2. **Doc-8 is the conformance harness, but is subordinate to its oracle.** Doc-8 *is* the gate that proves Doc-5/6/7 + Code conform, and passes its own Doc-8A **Appendix A** (`CHK-8-xxx`) before any per-suite freeze — yet it holds no authority over what it tests. A red conformance test means the implementation is wrong (fix it) or the corpus is defective (`[ESC-8-CORPUS]`, flag-and-halt) — never that the test should be relaxed. It coins no contract, route, field, permission slug, state, event, audit action, POLICY key, or expected value.

## Decisions proposed for ratification at structure freeze (R-set)

- **R1 — Doc-8 program shape: a metastandard + test-layer-partitioned realizations.** Doc-8 = **Doc-8A** (this test realization metastandard) + a set of realization documents partitioned **by test layer / verification discipline** (Doc-8B…8G below), each authored through the staged-freeze lifecycle and gated at freeze by Appendix A. **The partition axis departs from the Doc-5/Doc-6 backend-module letter map (B=M0…K=M9) by intent — and, like Doc-7, partitions by deliverable rather than by module:** a single conformance concern (e.g. the cross-tenant RLS gate, or the event-flow integration suite) spans many modules, so partitioning by test discipline keeps each verification concern whole and lets the shared harness freeze once. **The cross-cutting shared document (Doc-8B Test Foundation & Harness) freezes before the suites that consume it** — the same "shared-first" cadence as Doc-7B/7C. *(Alternative considered: per-backend-module partition parallel to Doc-5/6. Rejected — it would scatter a single cross-cutting gate (RLS byte-equivalence, event-flow, invariant firewall) across ten documents and duplicate the harness. Test-layer partition is the proposed shape; Board ratifies at freeze.)*
- **R2 — Realize-never-redecide; the frozen corpus + the realization contracts are the test oracle.** Every assertion in every Doc-8 suite traces **verbatim by pointer** to a frozen declaration — the domain *what* (Doc-2 §2/§3/§4/§5/§6/§8/§9; Doc-3 §12; Doc-4J/4L/4M; the 12 invariants + firewall) or a realization contract (Doc-5x field/error/pagination/idempotency; Doc-6x column/constraint/RLS policy; Doc-7x surface behavior). Doc-8 coins **no expected behavior**. Test infrastructure choices (harness shape, fixture/factory design, assertion style, mock topology) are the only *new* decisions, and they MUST NOT assert behavior absent from the corpus nor a rule stricter/looser than the owning contract. Gaps → `[ESC-8-*]` markers, escalated, never invented.
- **R3 — Test stack realization = TypeScript test tooling over the fixed Master-Architecture stack; the specific frameworks are an implementation choice ratified at freeze (the Doc-6 R3(b) precedent).** Realize tests against the already-decided runtime stack (Master Architecture / CLAUDE.md §2 — Next.js 15 / TypeScript ≥5 / Prisma / Supabase PostgreSQL / Inngest). The Master Architecture pins **no test framework**, so Doc-8 **coins none** — the concrete tooling is an implementation choice the Board ratifies at Doc-8A structure freeze (or carries via `[ESC-8-TOOLING]`), exactly as the one-Prisma-namespace-per-module choice was ratified at Doc-6A freeze. *(Candidate tooling, for Board ratification — not coined here: a TS unit/contract runner (e.g. Vitest/Jest), a browser e2e driver (e.g. Playwright), an RLS/SQL assertion path (e.g. pgTAP or transactional SQL fixtures against a Supabase test database), and an a11y/visual-regression path. The specific selection is `[ESC-8-TOOLING]` until ratified.)* No stack runtime element is coined or re-decided.
- **R4 — Conformance is Doc-8's defining function — the inverse of the Doc-6/Doc-7 "consistency-not-conformance" posture.** Where Doc-6/Doc-7 are *consistent-with* (not conformant-to) Doc-5A, **Doc-8 *is* the conformance harness** that gates the Doc-5/6/7 realizations and the eventual Code. But Doc-8 is **subordinate to its oracle**: it cannot change a contract, a schema, a surface, or a corpus declaration. A failing conformance test is dispositioned as **either** a code/realization defect (fix the implementation — Doc-8 wins, code conforms) **or** a genuine corpus/contract defect (`[ESC-8-CORPUS]` / `[ESC-8-API]` — flag-and-halt to the Board for a human-approved additive patch at the owning document). **It is never resolved by weakening the assertion.** The merge gate (R11) blocks code on any red conformance suite.
- **R5 — Test-data & tenancy realization (Users Act, Organizations Own — Invariant #5; One Module, One Owner — Invariant #7).** Every test seeds and acts through the **same active-org/tenant model** the runtime enforces: a server-resolved active-org context, never a client-trusted org ID. Any tenant-scoped suite seeds **≥2 organizations** (the minimum to prove isolation). Test data is created **through contracts/seed paths, never by hand-mutating another module's tables** — One Module, One Owner holds inside the test harness exactly as in production; a suite for module M may not write module N's tables directly. Fixtures realize the standard-column contract (Doc-6 §3) and the role/permission seeds (Doc-2 §7). No fixture invents a column or a role.
- **R6 — The RLS positive / negative / cross-tenant byte-equivalence gate (the load-bearing security gate Doc-6 explicitly delegates to Doc-8).** Realize the gate Doc-6A hands forward (`Doc-6A R8 / §4 / §11` — "the positive/negative/cross-tenant RLS test obligation is **Doc-8's**"): **positive** (an authorized actor under the active org sees exactly what the contract grants), **negative** (an unauthorized actor is blocked at both the app layer and the RLS backstop), **cross-tenant** (org A can never read/list/count org B's tenant-owned rows), and the **non-disclosure byte-equivalence** assertion (Invariant #11 / Golden Rule #5): a blacklisted/excluded vendor's responses, counts, analytics, notifications, logs, and errors are **byte-equivalent** to a non-matched vendor's — `buyer_vendor_statuses` and `link_suggestions` are never observable in any vendor-facing surface, view, count, or error. This is a **mandatory freeze-gate band** (Appendix A); it is realized in the persistence suite (Doc-8D) and asserted again at the UI in the FE suite (Doc-8G). No exclusion signal is ever permitted to leak.
- **R7 — Governance-signal firewall & the 12-invariant conformance suites (CLAUDE.md §4/§5; Doc-2).** Each of the **5 firewalled governance signals** and the **12 Core Invariants** has an enforcing suite that asserts the firewall holds: Financial Tier never raises Trust Score; Financial Tier does not affect Performance Score; Buyer Approved/Blacklisted never mutates platform-wide scores; no secondary signal dominates trust; no single signal dominates a matching decision; scores are auto-calculated under the System actor and never hand-edited; AI suggests but modules decide; capability (matrix of 4 flags) ≠ label; Financial Tier (capability) ≠ Subscription Plan (commercial); content ≠ presentation; the moat (governed M3 matching/routing + M4 post-award/CRM) is exercised. Bound by pointer to CLAUDE.md §4/§5 and the owning Doc-2/Doc-4 declarations. No signal, score, or invariant is redefined.
- **R8 — State-machine conformance (Doc-4M authority).** Every state machine (RFQ, quotation, verification, subscription, post-award documents, and every other Doc-4M machine) has a suite asserting: every **permitted** transition for the permitted actor in the permitted state succeeds; every **illegal** transition (wrong edge, wrong actor, wrong source state) is rejected; no state label or edge exists in the test that Doc-4M does not declare. Optimistic-UI reconciliation (Doc-7 R9) is asserted to converge to the server-authoritative state. No state or edge is coined.
- **R9 — Event & outbox integration conformance (Doc-2 §8 / Doc-4J catalog / Doc-4L flow map; M0-owned outbox).** Realize suites that assert: the **transactional write-plus-emit** atomicity (a business write and its `core.outbox_events` insert commit or roll back together — Doc-6 §7); event payloads conform to the Doc-4J authoritative catalog (zero coined events); cross-module effects propagate **only** via contracts/events and land in the **consuming** module's own schema (no cross-schema write); the outbox → Inngest dispatch lifecycle (`pending → dispatched → archived`) and the Doc-4L fan-out map are honored. No event is coined; no cross-module table access is introduced by a test.
- **R10 — Frontend & E2E conformance (Doc-7 surfaces).** Realize suites that assert the Doc-7 realizations: component behavior against the shared kit (Doc-7B); the **WCAG-AA + keyboard/focus** baseline (Doc-7 R11); visual-regression of presentation; end-to-end user journeys over the frozen Doc-5 surface (discovery → RFQ → quotation → award → post-award; account/billing; admin console); **per-value-field currency display, default BDT, never assumed** (Doc-2 §0.4); and the **UI non-disclosure byte-equivalence** re-assertion of R6 at the presentation layer (Doc-7 R8 — the Vendor workspace's load-bearing attestation). E2E uses only frozen Doc-5 contracts; it invents no endpoint.
- **R11 — Test isolation, determinism & CI merge-gating.** Every suite is **deterministic** (no wall-clock/random flakiness — a seedable clock and seedable/UUIDv7 ID source; the Doc-6 ID service is fed deterministically in tests), **isolated** (per-test transaction rollback or schema reset; no inter-test state bleed), and **hermetic** (no live external network — every out-of-wire boundary is mocked: Supabase Storage, Realtime, Resend email, PostHog, Inngest dispatch surface, and the M9 AI providers). The **CI gate blocks merge on any red conformance suite** (R4); flaky tolerance is zero — a non-deterministic test is a defect, not an accepted cost. AI-generated code clears the Doc-8 gate **plus** AI Coding Supervisor or human review before merge (CLAUDE.md §8); the gate never substitutes for the required review.
- **R12 — Out-of-test boundary (the verification analog of the out-of-wire / out-of-DB / out-of-frontend fence).** Tests own **no production state**: the test database is ephemeral; test doubles, fixtures, factories, and mocks are disposable and never a source of production truth; a test never writes to production data or a live external service. Mocked boundaries (Storage / Realtime / Resend / PostHog / Inngest / AI providers) are **simulated, never live**. A test may **observe** but never **author** the corpus contract it verifies. **Flag-and-halt if any test is proposed as a source of production truth, as the authoritative definition of a behavior, or as a backdoor that asserts behavior the corpus does not declare.**
