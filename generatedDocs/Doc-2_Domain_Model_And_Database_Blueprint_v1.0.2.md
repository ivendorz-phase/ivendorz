## 0. Global Conventions

These conventions apply to every table in this document and are not repeated per table.

### 0.1 Identifiers

- **Primary key:** every business entity uses `id UUIDv7`, generated via the Shared Kernel ID service. Identifiers never change.
- **Human reference:** customer-facing entities carry a separate `human_ref` (e.g., `ORG-2026-000001`, `RFQ-2026-000123`, `QTN-2026-000567`, `INV-2026-000891`, `DOC-2026-000891`): monotonic, gap-tolerant, never reused, year-scoped per entity type, generated by `core.id_sequences`. UUIDv7 remains the authoritative system identifier.

### 0.2 Standard Columns

| Column set | Applies to | Columns |
|---|---|---|
| Timestamps | all tables | `created_at`, `updated_at` |
| Actor stamps | all business tables | `created_by` (user_id), `updated_by` |
| Tenant boundary | all tenant-owned tables | `organization_id` (the owning organization; RLS anchor) |
| Soft delete | all soft-deletable tables | `deleted_at`, `deleted_by`, `delete_reason` |

- **Partial unique indexes:** every unique constraint on a soft-deletable table is `UNIQUE ... WHERE deleted_at IS NULL`.
- **Soft-deleted rows** are excluded from routing, matching, search, and default queries.

### 0.3 Module Schemas

One PostgreSQL schema per module. **No foreign keys cross schema boundaries.** Cross-module references are bare UUID columns (suffix `_ref` is not used; the column name states the entity, e.g., `vendor_profile_id`), validated by the owning module's service before save, reconciled by the periodic integrity audit job (orphan scans), and synchronized by events.

| Schema | Module |
|---|---|
| `core` | Module 0 ‚Äî Platform Core / Shared Kernel |
| `identity` | Module 1 ‚Äî Identity & Organization |
| `marketplace` | Module 2 ‚Äî Marketplace & Discovery |
| `rfq` | Module 3 ‚Äî RFQ Procurement Engine |
| `operations` | Module 4 ‚Äî Business Operations Engine |
| `trust` | Module 5 ‚Äî Trust & Verification |
| `communication` | Module 6 ‚Äî Communication |
| `billing` | Module 7 ‚Äî Monetization |
| `admin` | Module 8 ‚Äî Admin Operations |
| `ai` | Module 9 ‚Äî AI Layer |

### 0.4 Notation

- **(AR)** = aggregate root. Aggregates are transactional consistency boundaries: child entities are modified only through their root's service.
- **VO** = value object (no identity; stored embedded or as JSONB; immutable).
- **Tenancy classes:** `tenant-owned` (one organization's private data), `platform-owned` (iVendorz data), `shared` (cross-tenant by explicit grant or public by design), `derived` (regenerable, non-authoritative).
- **Currency:** all monetary amounts are `NUMERIC` with an explicit `currency` column defaulting to `'BDT'`.

## 2. Aggregate Design

### Module 0 ‚Äî core (no aggregates)

Infrastructure records only. `audit_records` and `outbox_events` are append-only streams; `system_configuration` and `feature_flags` are keyed configuration. No business aggregate exists here, by rule.

### Module 1 ‚Äî identity

| Aggregate | Root | Child Entities | Value Objects |
|---|---|---|---|
| User | `users` (AR) | ‚Äî | NotificationPreferences (JSONB), PersonalSettings |
| Organization | `organizations` (AR) | `memberships`, `organization_workflow_settings`, `buyer_profiles` | Address, ContactInfo, BrandAssets reference |
| Role | `roles` (AR) | `role_permissions` | ‚Äî |
| Delegation Grant | `delegation_grants` (AR) | ‚Äî | PermissionSet (JSONB array of slugs) |

- `permissions` is a platform-owned catalog (reference data), not an aggregate.
- Membership is a child of Organization (invited/removed through the organization), but carries its own state machine (¬ß5.2).
- **ASSUMPTION A-01:** `delegation_grants` are owned by Module 1, because grants are authorization infrastructure (they participate in the access formula alongside memberships and permissions). They reference `vendor_profile_id` cross-module by ID.

## 5. State Machines

Conventions: transitions are service-mediated, emit audit records, and (where listed in ¬ß8) emit domain events. `[guard]` = transition condition. Terminal states are bolded. Soft delete is orthogonal to these machines unless shown.

### 5.1 Organization

```
active ‚îÄ‚îÄsuspend [platform governance]‚îÄ‚îÄ‚ñ∂ suspended
suspended ‚îÄ‚îÄreinstate‚îÄ‚îÄ‚ñ∂ active
active|suspended ‚îÄ‚îÄsoft delete [Owner or admin; cascade ¬ß10-note]‚îÄ‚îÄ‚ñ∂ soft_deleted
soft_deleted ‚îÄ‚îÄrestore [restore-conflict rule: regenerate reused slugs]‚îÄ‚îÄ‚ñ∂ active
```

Guards: an organization must always have ‚â•1 active Owner (Last Owner Protection); ownership succession must run before owner removal/disablement. Cascade on soft delete: memberships ‚Üí soft-deleted; vendor profile ‚Üí suspended; RFQs ‚Üí archived; quotations ‚Üí preserved.

### 5.2 Membership

```
invited ‚îÄ‚îÄaccept‚îÄ‚îÄ‚ñ∂ pending ‚îÄ‚îÄverification complete‚îÄ‚îÄ‚ñ∂ active
active ‚îÄ‚îÄsuspend‚îÄ‚îÄ‚ñ∂ suspended ‚îÄ‚îÄreinstate‚îÄ‚îÄ‚ñ∂ active
active|suspended ‚îÄ‚îÄremove‚îÄ‚îÄ‚ñ∂ removed   (terminal; audit retained)
invited ‚îÄ‚îÄexpire/revoke‚îÄ‚îÄ‚ñ∂ removed
```

`invited` grants no access; `pending` grants no business access; only `active` participates in the access formula.

### 5.3 Vendor Profile

Two orthogonal dimensions ‚Äî claim lifecycle and operational status:

```
CLAIM:    seeded ‚îÄ‚îÄinvite sent‚îÄ‚îÄ‚ñ∂ invited ‚îÄ‚îÄvendor claims + logs in‚îÄ‚îÄ‚ñ∂ claimed
          claimed ‚îÄ‚îÄverification passed‚îÄ‚îÄ‚ñ∂ verified
          (direct registration creates the profile in claimed state)

STATUS:   active ‚áÑ suspended (admin/cascade)   active ‚îÄ‚îÄban‚îÄ‚îÄ‚ñ∂ banned (public banner)
          banned ‚îÄ‚îÄlift‚îÄ‚îÄ‚ñ∂ active
```

Guards: claim transition requires a controlling organization; the claim lifecycle applies **only to marketplace vendor profiles** (Architecture Patch v1.0.1, PATCH-02), never to private vendor records. Ownership transfer at any point triggers the Trust Protection Workflow (trust freeze ‚Üí compliance review ‚Üí reactivation). Soft-deleted profiles are excluded from routing/search.

### 5.4 RFQ

```

draft ‚îÄ‚îÄsubmit [approval required]‚îÄ‚îÄ‚ñ∂ pending_internal_approval
pending_internal_approval ‚îÄ‚îÄapprove [can_approve_rfq]‚îÄ‚îÄ‚ñ∂ submitted
pending_internal_approval ‚îÄ‚îÄreject‚îÄ‚îÄ‚ñ∂ draft
submitted ‚îÄ‚îÄmoderation pass‚îÄ‚îÄ‚ñ∂ under_review ‚îÄ‚îÄcleared‚îÄ‚îÄ‚ñ∂ matching
matching ‚îÄ‚îÄpipeline complete‚îÄ‚îÄ‚ñ∂ vendors_notified
vendors_notified ‚îÄ‚îÄfirst quotation‚îÄ‚îÄ‚ñ∂ quotations_received
quotations_received ‚îÄ‚îÄbuyer opens comparison‚îÄ‚îÄ‚ñ∂ buyer_reviewing
buyer_reviewing ‚îÄ‚îÄshortlist‚îÄ‚îÄ‚ñ∂ shortlisted
shortlisted ‚îÄ‚îÄaward‚îÄ‚îÄ‚ñ∂ **closed_won**        shortlisted ‚îÄ‚îÄno award‚îÄ‚îÄ‚ñ∂ **closed_lost**
vendors_notified|quotations_received|buyer_reviewing ‚îÄ‚îÄvalidity window lapses [system actor; window in system_configuration]‚îÄ‚îÄ‚ñ∂ **expired**

```

Guards: `estimated_value` is required at submission (**ASSUMPTION A-05** ‚Äî the tier gate is undefined without it; smallest assumption is mandatory-at-submit). Version immutability: once any quotation exists against version X, X is immutable; edits create version X+1 with a revision reason. Award records deal value for transaction intelligence. `closed_won` triggers engagement creation (Module 4) and performance inputs (Module 5).

### 5.5 Quotation

### 5.6 Verification (verification_record)

```

Verified-tier statuses are Pending Verification ¬∑ Verified ¬∑ Suspended ¬∑ Expired, with 24-month review; the architecture's `Declared` state is represented by the **absence** of a verified-tier record (one business state, one authoritative source ‚Äî the marketplace declared tier).

draft ‚îÄ‚îÄactivate‚îÄ‚îÄ‚ñ∂ active
active ‚îÄ‚îÄedit‚îÄ‚îÄ‚ñ∂ active (new template_version; prior versions retained)
active ‚îÄ‚îÄarchive‚îÄ‚îÄ‚ñ∂ archived       archived ‚îÄ‚îÄreactivate‚îÄ‚îÄ‚ñ∂ active
```

Never overwritten; generated documents record the template_version used.

### 5.10 Delegation Grant

```
draft ‚îÄ‚îÄgrant [granted_by must hold authority in controlling org]‚îÄ‚îÄ‚ñ∂ active
active ‚îÄ‚îÄsuspend‚îÄ‚îÄ‚ñ∂ suspended ‚îÄ‚îÄreinstate‚îÄ‚îÄ‚ñ∂ active
active|suspended ‚îÄ‚îÄrevoke‚îÄ‚îÄ‚ñ∂ **revoked**
active ‚îÄ‚îÄvalid_to passes‚îÄ‚îÄ‚ñ∂ **expired**
```

Guards: a grant never bypasses membership, permission, or organization validation; the acting user must hold an **active membership + relevant permission within the representative organization** AND the grant must be active and cover the requested action. The grant is a dual-party record: both organizations may read it, but only the **controlling organization** may create, suspend, revoke, or renew it (unless explicitly delegated). Revocation/expiry removes the derived `rfq_invitation_grantees` rows and visibility records for that representative (removals audited). Grants delegate authority; they do not create it.

### 10.11 Blueprint-Wide Enforcement Checklist (for Doc-3/4 and CI)

1. RLS policies on every tenant-owned table (`organization_id = active org`), with positive / negative / cross-tenant-denial tests in CI.
2. Partial unique indexes wherever uniqueness coexists with soft delete (`WHERE deleted_at IS NULL`).
3. No cross-schema FOREIGN KEY ‚Äî CI schema lint rejects them; write-time service validation + nightly integrity audit job scanning `orphan_vendor_ids, orphan_org_ids, orphan_rfq_ids, orphan_quotation_ids` (+ engagement and invitation orphans).
4. Outbox insert in the same transaction as every state-machine transition listed in ¬ß8.
5. The non-disclosure invariant verified by tests: a blacklisted vendor's API responses, counts, analytics, and logs are byte-equivalent to a non-matched vendor's.
6. `rfq_versions.is_immutable` enforced by trigger/service: UPDATE rejected once set.
7. Quotation uniqueness partial index (¬ß10.4) present before any quotation endpoint ships.
8. Human-reference allocation is concurrency-safe (row-locked `core.id_sequences`), gap-tolerant, never reused.
9. **RLS policies never depend on cross-schema ownership traversal.** Vendor-side access anchors are materialized: `rfq_invitation_grantees` rows (written at delivery by the RFQ Service) and explicit party columns (`quotations.controlling_organization_id`, `delegation_grants.controlling_organization_id` / `representative_organization_id`). Delegation-grant revocation removes the representative's grantee and visibility rows; removals are audited.
10. `matching_results` and invitation/lead creation paths verified by tests against the non-disclosure invariant: no gate-excluded vendor ever appears in derived tables, leads, or events.

## 11. Assumptions Register

| # | Assumption | Basis (smallest-possible rationale) |
|---|---|---|
| A-01 | `delegation_grants` owned by Module 1 (identity) | Grants are part of the access formula (authorization infrastructure); architecture names no owner |
| A-02 | Post-award container table named `operations.engagements`, one per Closed-Won award | Architecture defines the post-award chain but not its container; avoids collision with marketplace showcase projects |
| A-03 | Confirmed private‚Üîpublic link = columns on `private_vendor_records`; candidates in `admin.link_suggestions` | Architecture Patch v1.0.1 PATCH-05 makes Vendor Master Identity logical; link-not-merge requires only a reference + resolution queue |
| A-04 | `support_tickets` owned by Module 6 | Tickets are user‚Üîplatform communication; no other owner stated |
| A-05 | `estimated_value` mandatory at RFQ submission | Tier gate (pipeline) is undefined without a value; mandatory-at-submit is the smallest closure |
| A-06 | Minimal subscription state machine (pending_payment/active/expired); cancellation = `active + auto_renew=false` until period end | Architecture defines ownership + events only; a `cancelled` state that behaves like `active` would contradict itself |
| A-07 | Minimal advertisement state machine (draft/pending_review/scheduled/active/paused/completed/rejected) | Architecture names the module ownership and slots only |
| A-08 | `identity.roles` seeded platform defaults use org_id NULL rows; orgs may clone/customize bundles | "Roles are bundles" + org-configurable workflow; no schema stated |
| A-09 | `vendor_matching_attributes` implemented as a derived read-model rebuilt from change events | Architecture requires the RFQ engine to read marketplace/trust data via services; a denormalized contract surface is the minimal performant realization |
| A-10 | Formal decline recorded on `rfq_invitations.state = declined` (not a quotation state) | Performance framework defines response = quote OR formal decline; invitation is the natural carrier |
| A-11 | Organizations without an active subscription receive the Basic entitlement profile unless explicitly suspended or banned | Approved per Doc2_Patch_v1.0.1; entitlement resolution needs a defined fallback |

No assumption introduces a feature, changes ownership assigned by the architecture or patch, or weakens an invariant.
