## §4 — Buyer Private CRM Surface Realization (BC-OPS-1)

### 4.1 Endpoint Realization (§5.2/§5.3; inventory §2.2)
- Methods (all **User(buyer)**): `create_private_vendor` → `POST /operations/private_vendors` (`201`+`Location`); `update_private_vendor` → `PATCH …/{id}`; `archive_private_vendor` → `POST …/{id}/archive_private_vendor` (state command); `add_private_vendor_note` → `POST …/{id}/notes` (append child, `201`); `set_private_vendor_rating` / `set_buyer_vendor_status` / `clear_buyer_vendor_status` / `set_vendor_favorite` / `clear_vendor_favorite` / `confirm_vendor_link` / `dismiss_vendor_link` → `POST …/{id}/{command}` (domain commands); reads → `GET` (`get_private_vendor`, `list_private_vendors`, `get_buyer_supplier_relationship`).
- Inputs per §5.4: `{id}`=`UUIDv7` in path; Request-Contract fields in body; **no** prohibited input (`Doc-4A §9.7`).
- **Binds:** `Doc-5A §5.2/§5.3/§5.4/§5.5`; `Doc-4F §F4`.

### 4.2 Private-Vendor & Buyer-Vendor-Status Machines (Doc-2 §3.5 — Doc-4M index)
- The private-vendor record machine (`active ⇄ archived`) is realized as **legal transitions only** — `archive_private_vendor` drives **`active → archived`** (`Doc-2 §3.5`; `Doc-4F §F4.3`); **reactivation is not a Doc-4F contract — no `archived → active` edge is added** (no transition invented). Illegal source → `STATE` → `409`; stale `expected_revision` → `CONFLICT` → `409`.
- **`buyer_vendor_status` is append-only history** (Approved / Conditional / Blacklisted) — `set_buyer_vendor_status` appends a new status row; `clear_buyer_vendor_status` appends a cleared state; **the prior history is never overwritten** (`Doc-4F §F4`). This is the private, per-buyer governance signal — **never disclosed, never evented** (R5/R6).
- **Binds:** `Doc-2 §3.5`; `Doc-4M`; `Doc-4F §F4.3/§F4.4`.

### 4.3 Non-Disclosure (R5 — load-bearing) & Disclosure Scope
- **Every §4 read is `Buyer-Org-Private`** (§3.5): `get_private_vendor`, `list_private_vendors`, `get_buyer_supplier_relationship` return only the active buyer org's own CRM data. **No buyer-vendor-status or buyer-private CRM fact (records, notes, ratings, favorites, link status) is ever surfaced to the vendor or any other tenant** (Invariant #11). A cross-org or non-entitled read collapses to a uniform **`NOT_FOUND`** identical in status/body/timing to genuinely-absent (no side-channel). **No status event is emitted.** The only sanctioned CRM-status egress is the **out-of-wire** `read_crm_status_for_routing` (§9, RFQ-consumed — blacklist indistinguishable from non-match).
- **Binds:** `Doc-5A §6.3`; `Doc-4A §7.5`; `Doc-2 §10.11`; `Doc-4F §F4`; §3.5.

### 4.4 Top-Level `reference_id` (C-05) — Doc-5F nominated declaration point

- **Cross-cutting:** this declaration **applies equally to §5–§8** — §4 is the nominated declaration point; the obligation is uniform across every M4 caller-facing surface and is not restated per section.
- **Binds:** `Doc-4A §22.1 C-05`; `PATCH-D4A-C05-204`; `Doc-5A §6`; `Doc-4F §3`.

### 4.5 Authorization, Idempotency, Error & Audit (no events — R5)
- **Authorization** server-side via `check_permission` (§3.4, sole authority); §4 slugs (`can_manage_private_vendors`, …, Doc-2 §7) never wire inputs; gaps → `[ESC-OPS-SLUG]`. All §4 commands are **Buyer-side, buyer-org scope** (delegation not eligible — buyer's own CRM).
- Mutations `Idempotency: required` → **`Idempotency-Key`** (`operations.idempotency_dedup_window` — Doc-3 §12.2, **registered via Patch v1.4**); replay → cached original, no duplicate audit (`Doc-5A §9.7`). `update_private_vendor` optimistic concurrency (`expected_revision`); stale → `CONFLICT` → `409`. Error per `Doc-5A §6.2` (by pointer; codes `Doc-4F §F4`, `ops_`): `VALIDATION`→400, `AUTHORIZATION`→403 (else `404` collapse — §4.3/R5), `NOT_FOUND`→404, `STATE`→409, `CONFLICT`→409, `BUSINESS`→422.
- Mutations **audited** via Doc-4B `core.append_audit_record.v1`; CRM mutations not separately enumerated carry **`[ESC-OPS-AUDIT]`** (nearest Doc-2 §9 action; never invented). **Emits no domain event** (R5 — status never evented; audit writes still occur per §4.5 above). `list_private_vendors` cursor-paginated (page-size via `operations.list_page_size_max`; no offset — `CHK-5A-070/071`).
- **Binds:** `Doc-5A §6/§9`; `Doc-4F §F4`; Doc-2 §7/§9; `operations.*` (Patch v1.4), `[ESC-OPS-SLUG]`, `[ESC-OPS-AUDIT]`.

- Methods: `update_engagement_status` / `close_engagement` → `POST /operations/engagements/{id}/{command}`; `record_delivery` → `POST …/engagements/{id}/record_delivery`; `record_buyer_feedback` → `POST …/engagements/{id}/record_buyer_feedback`; `issue_engagement_document` → `POST …/engagements/{id}/engagement_documents` (**`202`** async — R9); `revise_engagement_document` → `POST …/engagement_documents/{id}/revise_engagement_document` (**`202`** async); `issue_trade_invoice` → `POST …/engagements/{id}/trade_invoices` (`201`+`Location`); `update_trade_invoice_status` → `POST …/trade_invoices/{id}/update_trade_invoice_status`; `record_payment` → `POST …/trade_invoices/{id}/payments` (`201`+`Location`); `confirm_payment` → `POST …/payment_records/{id}/confirm_payment`; reads → `GET` (`get_engagement`, `list_engagements`, `get_engagement_document`).
- **Binds:** `Doc-5A §5.2/§5.3/§10`; `Doc-4F §F5`.

### 5.2 Engagement, Trade-Invoice & Payment Machines (Doc-2 §3.5/§10.5 — Doc-4M index)
- Realized as **legal transitions only** (no transition invented): **engagement** `open → in_delivery → completed → closed` (`update_engagement_status` / `close_engagement`; `Doc-2 §3.5/§10.5`; the **`create_engagement_on_award` System consumer is out-of-wire §9/R7** — the only creator); **trade invoice** `issued → partially_paid → paid | disputed | cancelled` (`update_trade_invoice_status`); **payment record** `recorded → confirmed` (`record_payment` → `confirm_payment`). Illegal source → `STATE` → `409`; stale → `CONFLICT` → `409`. Engagement documents (LOI/PO/challan/WCC) are **versioned children, never overwritten** (`Doc-2 §10.5`).
- **Binds:** `Doc-2 §3.5/§10.5` (authoritative state edges); `Doc-4M` (cross-module state-machine index); `Doc-4F §F5`.

### 5.3 Money Boundary (R8 — records, not rails) & Async Commercial Documents (R9)
- **R8:** `issue_trade_invoice` / `update_trade_invoice_status` / `record_payment` / `confirm_payment` realize **document/record state transitions only** — **no funds movement, no escrow, no settlement, no payment-gateway operation** (`operations.trade_invoices` ≠ `billing.platform_invoices`; payment records have **no funds custody** — DF-6). **Flag-and-halt if a settlement/transfer surface is proposed.** Operations never derives Billing entitlement or payment state (`billing.platform_invoices` boundary — DF-6).
- **R9 (async, Doc-5A §10):** `issue_engagement_document` / `revise_engagement_document` create a versioned document child and **enqueue the out-of-wire `generate_document` job** (§9), returning **`202`** (accepted-then-processing). The status-resource Query **`get_generated_document`** (§7, BC-OPS-4) **is the canonical Doc-5A §10 polling resource and source of truth** (Structure FROZEN R9), returning **`ASYNC_PENDING`** while the file is pending and **never a fabricated outcome** (`Doc-5A §10.2/§10.3`; `CHK-5A-090/-092`); the immutable `template_version` used is recorded. (`get_engagement_document` is the engagement-document request record — also returns `ASYNC_PENDING` as a body-state field while the job runs; it is **not** the Doc-5A §10 polling designation — Pass-1 §2.6 / row 27.)
- **Binds:** `Doc-5A §10`; `Doc-4A §15`; `Doc-4F §F5/§F7.3`; DF-6; structure R8/R9.

### 5.4 Actor-Side Declarations (§3 binding rule — frozen-sourced) & Disclosure Scope
- Per the §3 per-command actor-side rule, each §5 command declares its side (**resolved exclusively from frozen Doc-4F §F5 actor metadata** — definitive §5.7 declaration per Pass-1 §0.4/§2.6; a per-command frozen-source citation supersedes a conflicting tentative Pass-1 inventory assignment). **Actor-side assignments are derived exclusively from Doc-4F actor metadata — no actor side is assigned by convention or assumption.** Per command: `update_engagement_status` / `close_engagement` = Either party (`Doc-4F §F5.2` Actor: `User (party)`, Scope: party org); `record_delivery` = Either party (`Doc-4F §F5.3` Actor: `User (party)`, Scope: party org); `issue_engagement_document` / `revise_engagement_document` = Either party (`Doc-4F §F5.4` Actor: `User (party)`, Scope: party org); `issue_trade_invoice` / `update_trade_invoice_status` = Either party (`Doc-4F §F5.5` Actor: `User (party)`, Scope: party org); `record_payment` / `confirm_payment` = Either party (`Doc-4F §F5.6` Actor: `User (party)`, Scope: party org); **`record_buyer_feedback` = Buyer-side only** (`Doc-4F §F5.7` Actor: `User (buyer-side party)`, Scope: buyer party org; delegation **not** eligible — buyer's own action). **§6B delegation is eligible vendor-side** for Either-party commands (a representative org acting for a vendor it does not control); slugs `can_manage_engagements` / `can_create_documents` (+`can_approve_po` for PO) / `can_record_payments` / `can_approve_payment` (Doc-2 §7).
- **Disclosure scope:** `get_engagement` / `list_engagements` / `get_engagement_document` → **`Shared-Engagement`** (party-scoped — buyer org sees its side, vendor controlling org sees its side; RLS shared-by-parties, `Doc-2 §6/§10.5`); §6B-eligible vendor-side representative read. No read widens beyond a party org.
- **Binds:** `Doc-4F §F5`; `Doc-2 §6/§7/§10.5`; `Doc-4A §6B`; §3.4/§3.5.

### 5.5 Events (outbox — R10), Idempotency, Error & Audit
- **Events (R10):** BC-OPS-2 domain events are bound by pointer to the **`Doc-2 §8` operations catalog** — catalog and payloads owned there, not restated here (`CHK-5A-103`). These events are emitted to the **M0 transactional outbox** by the realizing command (business write + event insert, one transaction — `Doc-4B`). Trust consumes the performance-input events idempotently (DF-4); notification dispatch is Communication-authored (DF-7). **No caller webhook/push** (`Doc-5A §11.3`); emission mechanism is §9.
- Mutations `Idempotency: required` (`operations.idempotency_dedup_window`); replay → cached original, **no re-emitted outbox event** (`Doc-5A §9.7`); optimistic concurrency (`expected_revision`); stale → `CONFLICT` → `409`. Error per `Doc-5A §6.2` (by pointer; codes `Doc-4F §F5`, `ops_`); `DEPENDENCY`→`503` (transient upstream unavailability, `retryable: true` — `Doc-5A §6.2` status map; `Doc-4F H.4` error-class definition). Top-level `reference_id` on every body-bearing response (§4.4). Audited via `core.append_audit_record.v1` (Engagement / Financial domains, Doc-2 §9); un-enumerated actions carry `[ESC-OPS-AUDIT]`. `list_engagements` cursor-paginated (`operations.list_page_size_max`; `CHK-5A-070/071`).
- **Binds:** `Doc-5A §6/§9/§11`; `Doc-4F §F5`; `Doc-2 §8/§9`; `operations.*` (Patch v1.4), `[ESC-OPS-AUDIT]`.
