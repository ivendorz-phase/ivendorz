## The gap

Doc-4B ֲ§B10 (`core.append_audit_record.v1`) mandates that the audit-write is an internal-service primitive **"Invoked-By: any module's mutating contract, within that contract's transaction (ֲ§17.1)"**, that it appends **"exactly one immutable row"**, and that audit-append failure **rolls the caller's transaction back** because **"audit is mandatory and atomic for mutations, ֲ§17.1"** (ֲ§B10; reinforced by ֲ§B11 O-7: the audit-write primitive is "atomic with each caller's mutation").

ֲ§B10 is **silent on the RLS posture** of that write. It says *who invokes* (any module's mutating contract, in-tx) and *what is owned* (M0), but not *how a caller running under a tenant request context ג€” `app.is_platform_staff = false`, an active `app.active_org`/`app.user_id` ג€” is admitted to append the row* at the database layer. The physical RLS posture is a downstream (Doc-6B) concern (CR9: Doc-6B realizes the table, not the write logic), and the as-realized Doc-6B ֲ§2.2 policy frames `core.audit_records` RLS as a **platform-staff disclosure/read backstop** (`USING(...)`-only, no `WITH CHECK`, app-layer authz primary). Because Postgres uses a policy's `USING` as its `WITH CHECK` when the latter is omitted, that single `USING`-only policy made **INSERT** staff-only as a *side effect* ג€” narrower than the frozen append-only intent (ESC-W2-AUDIT-RLS ֲ§6). Read literally against it, a tenant-context caller (`is_platform_staff = false`) would be denied at the DB layer ג€” yet ֲ§B10 requires exactly that caller to append in-transaction. This patch closes the silence by **recording** the RLS contract the ֲ§B10 obligation depends on; it changes **no contract behavior**. The **rationale** for the asymmetry it records ג€” broad write-admission, narrow read-disclosure ג€” is owned by **ADR-021** (proposed: *Audit-Records RLS Asymmetry*), referenced here by pointer, never restated.

## The additive clarification

Bound to ֲ§B10 / ֲ§17.1, additively (the frozen ֲ§B10 prose unchanged):

1. **Admission of the in-tx append.** A mutating contract running under a **tenant request context** satisfies its ֲ§B10 audit obligation because the append row is admitted at the DB layer under a **dedicated, context-bound, append-only `FOR INSERT` RLS policy** on `core.audit_records` ג€” realized in the companion **D3 Doc-6B patch** (the mechanism is bound by pointer; the SQL is **not** restated here). Per ESC-W2-AUDIT-RLS ֲ§7 = R-b, the effective INSERT check is *(platform-staff) OR (context-bound)* ג€” the new permissive `FOR INSERT` policy OR-combines with the untouched platform-staff policy (Postgres: permissive policies of the same command combine with `OR`; the addition only widens, never narrows). **Staff, System, drainer, admin, and bootstrap legs append under the platform-staff leg; tenant-context callers append under the context-bound leg** (row bound to the server-set `app.active_org` / `app.user_id`, fail-closed on unset GUC). Per R-b binding condition (2), the tenant-user leg is `organization_id = app.active_org` **AND** `actor_id = app.user_id` **AND** `actor_type = 'user'` (the `actor_type = 'user'` pin is **bound**, not advisory ג€” it is the Board's recommended defense-in-depth against a tenant caller forging a `admin`/`system`/`ai_agent` `actor_type` into the CR4ג€²-immutable trail; the frozen `core.ActorType` enum labels are **lowercase** ג€” `'user'`, not `'User'` ג€” see the D3 patch's load-bearing enum-label note). This is the same frozen R-b `WITH CHECK` idiom already in force on the M1/`identity` tenant tables (`memberships_insert`, `buyer_profiles_tenant`, `roles_write`) ג€” bound by pointer to the D3 patch; coined nowhere here.

2. **NULL-organization and System-actor rows are staff-leg-only (by design).** An audit row with `NULL` `organization_id` (legitimate for platform/System actions ג€” the column is nullable, an audit-context reference, never an RLS anchor: CR2) cannot satisfy the tenant leg (`NULL = app.active_org` ג†’ `NULL` ג†’ false) and is admitted **only** under the platform-staff leg. This mirrors the frozen `identity` `roles_write` policy, which deliberately omits a `NULL`-org write leg so system-bundle rows are staff-only to write. Every tenant-context mutation runs under a context where `app.active_org` is server-set (the M0/M1 server guard always sets it for tenant requests), so a tenant-context audit row never legitimately carries `NULL` org. **If a genuine non-staff, no-active-org-context, audit-producing path is ever discovered, it can satisfy neither leg and would block the mutation ג€” Flag-and-Halt (CLAUDE.md ֲ§11); do not widen the leg locally.**

3. **Read surface unchanged.** The audit **READ (SELECT) surface stays platform-staff-only** ג€” the existing `audit_records_platform_staff` backstop (Doc-6B ֲ§2.2 / CR2) is **untouched**; the D3 INSERT policy is INSERT-only. A tenant context that may *append* still may **not** *read* audit rows. `organization_id` remains an audit-context reference column, never an RLS read predicate (CR2).

4. **Immutability untouched.** UPDATE/DELETE remain governed solely by the CR4ג€² column-aware immutability triggers (Doc-6B ֲ§4.1: no-DELETE + immutable payload + set-once `archived_at`). The D3 patch adds **no** RLS allowance for UPDATE/DELETE. Redaction stays a **new audit row** (the separate `core.admin_redact_audit_field` contract / ֲ§B5), never an in-place edit.

5. **Atomicity preserved (ֲ§17.1).** The append stays the in-transaction audit insert on **the caller's executor** ג€” no separate connection, **no privilege elevation, no `SECURITY DEFINER`, no RLS bypass** (R-b explicitly rejects the elevation leg R-a and the `SECURITY DEFINER` leg R-c). Audit-append failure still rolls the caller's transaction back (ֲ§B10 `core_audit_append_failed`; ֲ§B11 O-7). The atomic, single-transaction guarantee of ֲ§B10/ֲ§17.1 is **carried, not altered**.

**Net:** the **contract** (in-tx append of exactly one immutable row on the caller's executor) is unchanged ג€” R-b removes the RLS denial, not the atomic write path. The concrete *realization*, however, is **not** a no-op: the as-built service path uses `db.auditRecord.create` (Prisma `INSERT ג€¦ RETURNING`), and under the staff-only SELECT surface a tenant-context `RETURNING` aborts the INSERT (Open note 1). So while the contract behavior is carried unchanged, the concrete realization **MUST** move from `create()` to an app-generated-UUIDv7 + non-`RETURNING` insert for the tenant leg to function. This patch only **records the RLS posture** the ֲ§B10/ֲ§17.1 obligation relies on, closing the gap where ֲ§B10 was silent about how a tenant-context caller is admitted to append; the realization change is a binding D4 constraint (Open note 1), not a contract change.

## What is unchanged

- **No contract behavior change.** Signature, Effect ("append exactly one immutable row"), Invoked-By (in caller's tx, ֲ§17.1), idempotency (one record per safe replay), error codes, and Audit-Required/Events-Produced of ֲ§B10 are all unchanged.
- **No privilege elevation; no `SECURITY DEFINER`.** R-b explicitly rejects the elevation leg (R-a) and the `SECURITY DEFINER` leg (R-c). The append stays on the caller's executor under the caller's context.
- **No read-surface widening at the contract level.** Audit READ stays platform-staff-only (CR2 / Doc-6B ֲ§2.2). (The D4 `RETURNING` constraint in Open note 1 exists precisely so the read surface is **not** widened to accommodate the read-back.)
- **No immutability change.** CR4ג€² triggers remain the sole guard for UPDATE/DELETE; redaction-as-new-row stands.
- **No architecture/ownership/governance-signal/state/event change; no new permission or POLICY key.** M0 still owns the audit-write; the five firewalled signals are untouched (audit is M0 infrastructure, not one of the five signals); "audit ג‰  events" (ֲ§17.6) stands.
- **No tenant-table RLS weakened.** The context-bound leg only admits truthfully-attributed audit rows; it widens nothing on `identity` or any other tenant table (R-b condition 6).
- **Frozen text not reopened.** `Doc-4B_Content_v1.0_PassB.md` ֲ§B10 is not edited in place; Doc-4A is not reopened; Doc-6B `CR2`/`CR4ג€²`/`CR9` and ֲ§2.2 are referenced, never re-authored.

## Open implementation notes (D4/D5)

These are realization concerns for the companion D3 (Doc-6B policy) and D4 (the `core.contracts` concrete facade) work ג€” recorded so they are not lost, **not** new contract mandates. They must be honored additively; **if any cannot be, Flag-and-Halt (CLAUDE.md ֲ§11), do not resolve locally.**

1. **`RETURNING`/SELECT-policy coupling ג€” BINDING D4 precondition (not merely advisory).** Prisma `db.auditRecord.create` always emits `INSERT ג€¦ RETURNING`. PostgreSQL forces every `RETURNING` row through the table's **SELECT** policy; a row that does not satisfy it **raises `ERROR: new row violates row-level security policy` (SQLSTATE 42501)** ג€” rows are never silently dropped (verified verbatim against the PG 15ג€“18 `CREATE POLICY` docs). With the audit SELECT surface staying **platform-staff-only** (binding condition (3)), a row inserted under a tenant context ג€” even one that passes the D3 INSERT `WITH CHECK` ג€” **fails the SELECT check on `RETURNING`, aborting the whole INSERT and rolling back the caller's business transaction.** Net: if D3 ships without D4 fixing the read-back, every tenant-code audit append fails ג€” the exact opposite of R-b's "tenant may append." This means R-b's tenant leg **cannot** be realized through a `RETURNING`-emitting `create` without either (a) widening the SELECT surface ג€” which **violates binding condition (3)** (staff-only read) and is therefore **not** permitted additively ג€” or (b) **removing the read-back** so the SELECT policy is never consulted. The condition-honoring realization is **(b): app-generated UUIDv7 + a non-`RETURNING` insert** (e.g. `$executeRaw`/`$executeRawUnsafe` INSERT) on the caller's executor ג€” M0 already mints the `auditId` app-side via `uuidv7()`, so no DB-returned key is needed; this also realigns the insert onto the `CoreServiceExecutor` contract surface (which exposes only `$queryRawUnsafe`, not the Prisma model delegate). **This is a binding precondition: D3 and D4 form a single linked deploy unit ג€” the D3 INSERT policy MUST NOT be deployed to any environment carrying tenant traffic until D4 has replaced `db.auditRecord.create` with the non-`RETURNING` insert.** Do **not** unblock by widening the audit read surface or by elevating privilege / `SECURITY DEFINER` ג€” both are rejected by R-b and by binding conditions (3) and (6). D4 must reconcile the audit-service insert with this constraint before shipping. This sequencing is elevated to a normative **Deployment Constraint** in the D3 patch (*the D3 migration SHALL NOT deploy to tenant-traffic environments until D4 has eliminated `INSERT ג€¦ RETURNING`*); **D3 + D4 are a single linked deploy unit.**
2. **Partition propagation + conformance (binding condition (5)).** RLS does not inherit to partitions. The D3 INSERT policy MUST be created on the parent `core.audit_records`, on `core.audit_records_default`, and on **every** operationally-provisioned monthly partition ג€” alongside the existing platform-staff policy. Partition provisioning MUST apply **both** policies atomically (a single provisioning template/function), so no partition ever carries the parent policy alone (a direct-to-partition append bypass). The partition-provisioning runbook is updated accordingly, and an Appendix-A-style **conformance check** asserts every `audit_records` partition carries the expected policy set (staff `USING` + context-bound `FOR INSERT`). This is a D3/operational obligation; recorded here only so the ֲ§B10 contract reader knows the append's admission depends on it.
3. **Fail-closed (binding condition (4)).** Unset GUC ג†’ `current_setting(..., true)` returns `NULL` ג†’ predicate false. The context-bound leg must never silently widen when `app.active_org` / `app.user_id` are unset.
4. **D4 facade (mirror `allocateHumanReference`).** The concrete `appendAuditRecord` facade is **currently a type-only export** in `core/contracts/services.ts` (the `AppendAuditRecord` type + a `CoreServices` field exist; no `export const`). D4 binds it mirroring the `allocateHumanReference` concrete-facade pattern ג€” `import { appendAuditRecord as appendAuditRecordImpl } from "../infrastructure";` then `export const appendAuditRecord: AppendAuditRecord = appendAuditRecordImpl;` (same-module `core/contracts` ג†’ `core/infrastructure` binding; contracts-only cross-module; coins nothing) ג€” and must keep passing `?? null` for the nullable `actor_id` / `organization_id` (both columns are nullable). This patch records the RLS posture only; the facade binding is D4's deliverable.
5. **D5 conformance-test cases (from ESC-W2-AUDIT-RLS ֲ§7 sequencing).** D4/D5 must prove, against the realized policy:
   - **(a) Tenant append succeeds atomically.** A **tenant-context** business write (`app.is_platform_staff = false`, `app.active_org`/`app.user_id` set, `actor_type = 'user'`) appends **exactly one** audit row **atomically** with the business write (same transaction; both commit or both roll back).
   - **(b) No staff-privilege leak.** The append introduces **no** elevation of `app.is_platform_staff` visible to any later write in the same transaction (verifies R-b's "no elevation" over R-a).
   - **(c) Forged/cross-tenant insert rejected.** A tenant-context insert that forges `organization_id ג‰  app.active_org`, or `actor_id ג‰  app.user_id`, or `actor_type ג‰  'user'`, is **rejected** by the `WITH CHECK` (SQLSTATE 42501).
   - **(d) Read stays staff-only.** A tenant context **cannot** `SELECT` any audit row (including rows it just appended) ג€” the staff-only read backstop is intact.
   - **(e) Staff/System/NULL-org append.** A platform-staff/System/drainer/bootstrap context (`is_platform_staff = true`) appends successfully, including a `NULL`-`organization_id` row, under the staff leg.
   - **(f) No `RETURNING` abort.** The realized audit insert does not abort under the staff-only SELECT surface ג€” i.e. the non-`RETURNING` realization of Open note 1 is in place (a regression guard against reintroducing `create()`).
