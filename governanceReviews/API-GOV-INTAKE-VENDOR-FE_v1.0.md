# API Governance Board — Intake: iVendorz Vendor Workspace contract questions

## Cover

**Purpose.** A single consolidated intake of every *contract* question that the Vendor Workspace design companion (`vendor_planning_and_design.md`, v0.9-rc — coverage complete, freeze WITHHELD) raised during Track-2/3 corpus verification. These are **additive-contract** or **corpus-reconciliation** decisions owned by **module teams** under §7 rank 5 (API Governance Board) and, for one item, **human-only corpus reconciliation** (§7 ranks 0–1, Flag-and-Halt). They do **not** block on the human Architecture Board for the additive-contract subset.

**Provenance.** Track-2 (caller-facing wired-contract verification) + Track-3 (corpus-anchor verification) over the frozen corpus on disk at `e:\Projects\iVendorz\generatedDocs\`. Every anchor below was re-confirmed at `doc:line`.

**NONE of these is a BLOCKER.** The companion ships with a graceful, corpus-conformant fallback for each. This intake exists to *raise* the contract questions (Raise ≠ Accept, CLAUDE.md §13) — the Board adjudicates and rules; module teams implement only validated decisions.

**Standing CONSTRAINTS every new vendor-facing contract MUST inherit** (binding, non-negotiable; a proposed contract that violates any is out of order):

1. **Wired-vs-out-of-wire boundary (Doc-7A §3.7).** The frontend (incl. its server layer) consumes only the **caller-facing wired** Doc-5 subset; **internal-service / out-of-wire** contracts (e.g. `check_permission`, `resolve_entitlements`, and any contract typed `internal-service`) are **never frontend-callable** (`Doc-7A_SERIES_FROZEN_v1.0.md:34`). Any participation/derived read consumed by the client must be a **new wired read**, never the existing internal-service contract.
2. **Cursor pagination / no leaking totals.** All list reads are `page_size` + `page_token` (page-token cursor, Doc-4A §22.3); no contract returns a grand total or cross-tenant aggregate (confirmed across BC-OPS-2/3/4 read schemas, e.g. `Doc-4F_PassB_Part3_BC-OPS-3_FROZEN.md:210,212`).
3. **Byte-equivalence non-disclosure (Doc-7A R8; CHK-7-040 / CHK-5A-073).** A vendor must not infer competitor existence/count, routing outcome, ranking/matching signals, or hidden RFQ decisions. Any count read uses **received-only denominators** — received-invitations / submitted-quotations / own-org `vendor_leads`-by-stage — never a routed/eligible/total population.
4. **Firewall (CLAUDE.md §4).** No new read may expose or mutate a governance signal across the firewall; vendor-private lead state is CRM-only and is **not** a governance signal (`Doc-4F_PassB_Part3_BC-OPS-3_FROZEN.md:148`).
5. **`file_ref`/`storage_ref`-only.** Document contracts carry refs, never in-row blobs (Doc-2 §10.5; `Doc-4F_PassB_Part4_BC-OPS-4_FROZEN.md` H.9).

## Per-item detail (MAJOR items)

**`ESC-7-API` #1 — received-only pipeline count (M3 and/or M4).** The Vendor Dashboard wants headline numbers ("N invitations", "M quotations", "K leads at stage X"). No frozen contract returns an aggregate; every vendor-leg read is a cursor list. The **owner is ambiguous by design**: invitation/quotation counts are M3-leg facts, but the cleanest source for a leads-by-stage tally is M4's own-org `vendor_leads` (the caller's own tenant rows, no cross-tenant exposure). Any count contract is bound to **received-only denominators** — it may count what the vendor *received* (delivered invitations) or *submitted* (quotations) or *owns* (its own leads-by-stage), and nothing about the routed/eligible/total population, preserving CHK-7-040 / CHK-5A-073 byte-equivalence. Fallback (shipping): the Dashboard shows **non-numeric "view" links** into each list, no badges. Recommendation: **decline a dedicated count contract for v1**; if numbers are wanted later, scope a single own-org `vendor_leads`-by-stage count owned by **M4**, since it never touches the routing population. Priority **Med**.

**`ESC-7G-Q-DRAFT` — server-side quotation draft persistence.** The frozen model creates the quotation **at submit**: `rfq.submit_quotation.v1` §6 is explicit that the allowed source is `draft` "(or first submit creating the quotation)" and the partial-unique index `UNIQUE(rfq_id, vendor_profile_id) WHERE state IN ('draft','submitted')` already contemplates a `draft` row (`Doc-4E_PassB_Part4_v1.0_FROZEN.md` §6; `Doc-2:765`). So a `draft` *state* exists in the data model, but **no caller-facing draft-write/draft-read contract** is frozen. This reopens the companion's earlier `[ESC-7G-A6]` "autosave resolved" — it was resolved *as client-local*, and the question is whether to harden it server-side. Question for the Board: additive `rfq.save_quotation_draft.v1` / draft-read pair, or accept client-local drafts. If additive: vendor controlling-org scope; **must not** emit `QuotationSubmitted` (that event is owned by submit, §8); the existing partial-unique governs one-active-per-vendor. Fallback (shipping): client-local "Saved on this device". Recommendation: **defer/decline for v1** (client-local is conformant and lossless within a device); revisit if cross-device draft continuity becomes a product requirement. Priority **Med**.

**`ESC-7-API` #2 — wired participation read (M1).** The derived Platform Participation label (Buyer/Vendor/Hybrid) is computed from the org's `participation_flags` (`has_buyer_profile`/`has_vendor_profile` derived, Doc-2 §10.2; `Doc-2:718`). The only contract carrying it, `identity.get_organization.v1`, is typed **internal-service** (`Doc-4C_Content_v1.0_PassB.md:124` "21.3 Query (internal-service) · Actor: internal-service"; catalog row `:751`). Doc-7A §3.7 forbids the frontend consuming internal-service contracts (`Doc-7A_SERIES_FROZEN_v1.0.md:34`). There is **no clean fallback** — without a wired read the UI simply cannot render a participation label, which degrades the Hybrid experience (server-resolved active org, Hybrid mounts both — Doc-7A R6). Recommendation: this is the **highest-value additive read** — propose a thin **wired** `identity.get_active_org_participation.v1` (or equivalent caller-facing projection) returning the derived flags/label for the server-resolved active org only, no membership detail. Priority **High**.

**`ESC-7G-ENG-01` — `rfq_id` on the engagement (M4).** Re-confirmed: `ops.get_engagement.v1` projects `engagement{ engagement_id, human_ref, status, buyer_organization_id, vendor_profile_id, vendor_controlling_org_id, award_value_snapshot, currency }` and **no `rfq_id` and no RFQ/quotation human-ref** (`Doc-4F_PassB_Part2_BC-OPS-2_FROZEN.md:462`). The underlying `operations.engagements.rfq_id` column **exists** in Doc-2 §10 (`Doc-2:780`), so projecting it is a pure additive read extension — no new column, no cross-module FK (the value is a bare UUID the vendor already owns context for). Fallback (shipping): `"[pending projection]"` where the engagement would link back to its originating RFQ. Recommendation: **approve the additive projection** of `rfq_id` on `ops.get_engagement.v1` (§F5.8) — lowest-risk, highest-utility of the M4 items; it restores the post-award→RFQ breadcrumb. Priority **High**.

**`ESC-7G-ENG-03` — engagement document enumeration (M4).** There is no `list_engagement_documents`. `ops.get_engagement_document.v1` fetches **one** document by `document_id` (`…Part2…FROZEN.md:462`), and `ops.list_generated_documents.v1` (BC-OPS-4) enumerates **rendered artifacts** (`generated_documents`, filterable `{ doc_kind?, source_entity_id? }` — `Doc-4F_PassB_Part4_BC-OPS-4_FROZEN.md:275`), which is a *different aggregate* from the BC-OPS-2 engagement-document chain (the boundary is explicitly held: BC-OPS-2 only references `template_version_id`; no ownership overlap). So the vendor cannot list the LOI/PO/challan/invoice/WCC **record set** for one engagement from a single contract. Two clean options: (a) additively extend `ops.get_engagement.v1` to return **child document refs** (refs only, `storage_ref`/`document_id`), or (b) add a party-scoped `ops.list_engagement_documents.v1` (cursor). Note: `list_generated_documents`'s `source_entity_id` filter could *partially* enumerate artifacts tied to an engagement, but it returns rendered artifacts, not the BC-OPS-2 records, so it does not satisfy the gap. Fallback (shipping): **E3 per-kind enumeration is build-blocked**. Recommendation: prefer **option (a)** — additive child-ref projection on `get_engagement` — as it stays inside the BC-OPS-2 aggregate and avoids a new list endpoint. Priority **High**.
